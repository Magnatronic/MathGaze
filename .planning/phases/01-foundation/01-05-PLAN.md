1. Verify MathGaze.csproj contains all required publish properties. Read the file and confirm these
   properties are present in the first PropertyGroup:
   - `<RuntimeIdentifier>win-x64</RuntimeIdentifier>`
   - `<SelfContained>true</SelfContained>`
   - `<PublishSingleFile>true</PublishSingleFile>`
   - `<IncludeNativeLibrariesForSelfExtract>true</IncludeNativeLibrariesForSelfExtract>`
   - `<DocnetRuntime>win-x64</DocnetRuntime>`

   If any are missing, add them now before proceeding.

2. Run the publish command:
```bash
dotnet publish MathGaze/MathGaze.csproj -c Release -r win-x64 --self-contained true \
  -p:PublishSingleFile=true \
  -p:IncludeNativeLibrariesForSelfExtract=true \
  -o publish/
```

3. Inspect the publish output directory:
```bash
ls publish/
```

Expected outcome:
- `MathGaze.exe` — present (the single-file EXE)
- `pdfium.dll` — must NOT be present as a loose file (it should be bundled)
- No other loose DLLs or framework files

If `pdfium.dll` is present as a loose file in the publish output, the native DLL is NOT bundled.
Mitigation: add the following to MathGaze.csproj to force bundling:
```xml
<ItemGroup>
  <None Include="$(USERPROFILE)\.nuget\packages\docnet.core\2.6.0\runtimes\win-x64\native\pdfium.dll">
    <CopyToOutputDirectory>Always</CopyToOutputDirectory>
    <ExcludeFromSingleFile>false</ExcludeFromSingleFile>
  </None>
</ItemGroup>
```
If this approach does not work, fall back: accept pdfium.dll as a companion file alongside the EXE
(i.e. publish as two files: MathGaze.exe + pdfium.dll). Document this in the SUMMARY as a known
limitation. The CORE-04 requirement is "no admin install, no pre-installed runtime" — a companion DLL
alongside the EXE still meets this requirement (xcopy-deployable from USB stick).

4. Verify EXE file size is reasonable:
   - Expected: 80–200 MB (self-contained .NET 9 + WPF + SkiaSharp + bundled PDFium)
   - If under 10 MB: not self-contained — the .NET runtime is missing from the bundle
   - If over 400 MB: investigate unexpected large files in the publish output

<threat_model>
  <threat id="T-05-01" likelihood="medium" impact="medium">
    <description>Elevation of Privilege: PDFium DLL extraction to %TEMP%\.net\ on first run. An attacker who has write access to %TEMP% before first launch could pre-plant a malicious pdfium.dll at the content-hash path, causing the app to load it instead of the bundled one.</description>
    <mitigation>.NET's single-file extraction uses a content-addressable path derived from the bundle hash: %TEMP%\.net\[app-name]\[bundle-hash]\. An attacker would need to know the exact bundle hash before the app runs, and have write access to %TEMP% — which is a per-user directory. If they have write access to the current user's %TEMP%, they can already run code as that user. Risk is accepted: it is below the practical threat model for a school tool. No additional code change required.</mitigation>
  </threat>
  <threat id="T-05-02" likelihood="low" impact="low">
    <description>Information Disclosure: the published EXE can be decompiled (IL is not obfuscated). Source code and structure are recoverable. This is an open-source-style deployment.</description>
    <mitigation>Accept. MathGaze is a student tool, not a security product. No secrets are embedded in the EXE (no API keys, no credentials). Decompilation reveals UI code, not anything sensitive.</mitigation>
  </threat>
</threat_model>
