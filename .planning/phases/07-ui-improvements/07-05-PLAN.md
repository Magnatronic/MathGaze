<task type="auto">
  <name>Task 1: Convert ToolTileStyle and IconButtonStyle style-level setters to DynamicResource in AppStyles.xaml</name>
  <files>MathGaze/Styles/AppStyles.xaml</files>
  <read_first>
    - MathGaze/Styles/AppStyles.xaml — read entire file before any edits; confirm current line numbers for the four StaticResource setters in ToolTileStyle and IconButtonStyle
    - MathGaze/Styles/Themes/Light.xaml — confirm BrushBorder, BrushInk, BrushInk2 are defined
    - MathGaze/Styles/Themes/Dark.xaml — confirm same keys are defined with dark values
  </read_first>
  <action>
    Root cause: WPF style-level `<Setter>` blocks bind the property value at element creation time. When the ResourceDictionary is swapped (ApplyTheme), elements that used `StaticResource` keep the old brush object — they never re-look up the key. Only `DynamicResource` in a style `<Setter>` creates a live binding that re-resolves on ResourceDictionary changes.

    In AppStyles.xaml, make exactly FOUR changes:

    1. ToolTileStyle, Setter for BorderBrush (currently `{StaticResource BrushBorder}`):
       Change to: `Value="{DynamicResource BrushBorder}"`

    2. ToolTileStyle, Setter for Foreground (currently `{StaticResource BrushInk}`):
       Change to: `Value="{DynamicResource BrushInk}"`

    3. IconButtonStyle, Setter for BorderBrush (currently `{StaticResource BrushBorder}`):
       Change to: `Value="{DynamicResource BrushBorder}"`

    4. IconButtonStyle, Setter for Foreground (currently `{StaticResource BrushInk2}`):
       Change to: `Value="{DynamicResource BrushInk2}"`

    Do NOT change:
    - DeleteButtonStyle (uses hardcoded hex colours #CC2020 etc. — not brush keys, no change needed)
    - RailButtonStyle and StepButtonStyle (already DynamicResource from Plan 01 — do NOT touch)
    - `{StaticResource BrushAccent}` anywhere — BrushAccent is identical in both themes; StaticResource is fine
    - `{StaticResource BrushTransparent}` anywhere — same as BrushAccent, identical in both themes
    - `{StaticResource FontBody}` or `{StaticResource FontMono}` — fonts, not theme brushes, no change

    After making changes, scan the file for any remaining `{StaticResource Brush` (excluding BrushAccent and BrushTransparent) to confirm none remain in style-level setters.
  </action>
  <verify>

  </verify>

<task type="auto">
  <name>Task 2: Suppress native scrollbar in object list panel by wrapping ItemsControl in explicit ScrollViewer</name>
  <files>MathGaze/Views/RightRail.xaml</files>
  <read_first>
    - MathGaze/Views/RightRail.xaml — read the full file; locate the ItemsControl for ObjectList (around line 97)
  </read_first>
  <action>
    Root cause: WPF's default `ItemsControl` control template wraps its content panel in a `ScrollViewer`. When content overflows, the default `ScrollBar` ControlTemplate (part of WPF's Aero/Fluent theme) renders the scrollbar thumb as an `Ellipse` shape. In dark mode, this ellipse is large, dark, and visually dominant.

    Preferred fix: wrap the `ItemsControl` in an explicit `ScrollViewer` with native scrollbars hidden. The custom ScrollRail buttons handle all page navigation already; the object list is short (one row per object on page) and will rarely overflow the available rail height.

    In RightRail.xaml, locate this block inside the ObjectListPanel (HasObjectList) Border:
    ```xml
    <!-- Object list (only when objects exist) -->
    <ItemsControl ItemsSource="{Binding ObjectList}">
      <ItemsControl.Style>...</ItemsControl.Style>
      <ItemsControl.ItemTemplate>...</ItemsControl.ItemTemplate>
    </ItemsControl>
    ```

    Wrap the ItemsControl in a ScrollViewer with a MaxHeight to prevent the object list from
    pushing the ClearPage/Undo buttons off-screen. Use a MaxHeight of 260 (enough for ~5 objects
    at 48px each + margins) — if more objects exist the list scrolls internally but uses no native
    WPF scrollbar chrome:

    ```xml
    <ScrollViewer VerticalScrollBarVisibility="Hidden"
                  HorizontalScrollBarVisibility="Disabled"
                  MaxHeight="260"
                  CanContentScroll="False">
      <ItemsControl ItemsSource="{Binding ObjectList}">
        <ItemsControl.Style>...</ItemsControl.Style>
        <ItemsControl.ItemTemplate>...</ItemsControl.ItemTemplate>
      </ItemsControl>
    </ScrollViewer>
    ```

    The `VerticalScrollBarVisibility="Hidden"` hides the native ellipse-thumb scrollbar.
    The `MaxHeight="260"` prevents layout overflow. The ScrollViewer still supports mouse wheel
    scrolling when there are many objects — it just doesn't show native chrome.

    Do NOT modify anything else in RightRail.xaml — no layout changes, no brush changes,
    no button size changes. Only wrap the existing ItemsControl.
  </action>
  <verify>
    <automated>dotnet build MathGaze/MathGaze.csproj 2>&1 | Select-String "error" | Select-String -NotMatch "NU1701"</automated>
  </verify>
  <done>

    - Build succeeds with 0 errors
    - No `<ScrollViewer>` with default VerticalScrollBarVisibility (which defaults to Visible/Auto and would show native chrome) is present anywhere else in the file
  </done>
