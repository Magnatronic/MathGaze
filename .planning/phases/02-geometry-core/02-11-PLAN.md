<objective>
Fix GAP-11: first click at a position places the dot inaccurately; second click at the same position places correctly.

Root cause: a ZoomFactor race between `SetCanvasSize` and the first user click.

The sequence that breaks:
1. PDF opens. `OpenFileAsync` calls `OnDocumentOpenedAsync` → `LoadCurrentPageAsync` with `ZoomFactor=1.0` (fit-page hasn't run yet).
2. `SetCanvasSize` fires (canvas is laid out). It calls `_mainVm.OnCanvasSizeChanged()` → `ApplyFitPage()` → `ZoomFactor` is updated to the fit-page value (e.g. 0.75). This is synchronous on the UI thread.
3. `ApplyFitPage` → `ZoomFactor` change → `PropertyChanged` → `OnMainViewModelPropertyChanged` → `LoadCurrentPageAsync` (async, fires in background). The async bitmap load has NOT completed yet.
4. `EnsureCoordinateMapper` was last called from `Paint()` BEFORE step 2 (when zoom was 1.0). So `_coordinateMapper` exists but has `_zoomFactor=1.0`.
5. `SetCanvasSize` returns (synchronous part done). No further synchronous call to `EnsureCoordinateMapper`.
6. User clicks before the async bitmap reload completes. `HandleCanvasClick` → `EnsureCoordinateMapper()`. Mapper is non-null → `Update()` called with CURRENT `_mainVm.ZoomFactor` (the fit-page value). Click is correctly mapped.

Wait — if `Update()` uses the CURRENT `ZoomFactor`, the first click should already be correct. Re-examining `EnsureCoordinateMapper`: it always calls `Update()` if the mapper is non-null, using current `_mainVm.ZoomFactor`. So the mapper is correct at click time.

The actual race: `_coordinateMapper` can be null at first-click time. `EnsureCoordinateMapper` at line 305 returns early if `_canvasWidthPx == 0`. The call sequence where this happens:

- App starts. `SizeChanged` fires during layout BEFORE DataContext arrives. `_vm` is null → `ReportCanvasSize` is a no-op → `_canvasWidthPx` stays 0.
- DataContext arrives → `WireViewModel` → checks `ActualWidth > 0 && ActualHeight > 0`. At this moment, if ActualWidth is 0 (layout not complete), `ReportCanvasSize` is NOT called.
- PDF opens → `OnDocumentOpenedAsync` → `LoadCurrentPageAsync` → `_canvasWidthPx == 0` → early return, no bitmap.
- FitPage → `ZoomFactor` changes → `LoadCurrentPageAsync` → still early return (canvas size still 0).
- Eventually another `SizeChanged` fires (window resize, maximize) → `ReportCanvasSize` → `SetCanvasSize` with real dimensions → `LoadCurrentPageAsync` → bitmap loads → Paint → `EnsureCoordinateMapper` creates mapper. NOW things work.

But in this path there's NO "first wrong, second correct at same position."

The confirmed bug from UAT: "First click at a position places the dot at the wrong location; second click at the EXACT SAME position places correctly." This means BOTH clicks produce a dot, but at different positions — not a null/rejected click.

The only way this can happen: `EnsureCoordinateMapper()` produces DIFFERENT mappers for click 1 vs click 2 at the same screen position, with the same `_mainVm.ZoomFactor` and `_dpiScale`. The only field that could differ: `_canvasWidthPx` or `_canvasHeightPx` (which affect `originX` centering).

The race: `SetCanvasSize` is called from `ReportCanvasSize` which is called from `OnCanvasSizeChanged` when the window resizes. `SetCanvasSize` → `_mainVm.OnCanvasSizeChanged()` (synchronous) → `ApplyFitPage()` → may set `ZoomFactor` → triggers `OnMainViewModelPropertyChanged` → calls `_ = LoadCurrentPageAsync()` (fire-and-forget async). Then `SetCanvasSize` continues: calls `_ = LoadCurrentPageAsync()` again. Both are fire-and-forget.

But here's the key: `OnCanvasSizeChanged()` → `ApplyFitPage()` is on the UI thread. `ZoomFactor` setter → `PropertyChanged` → `OnMainViewModelPropertyChanged` in PdfCanvasViewModel — but PdfCanvasViewModel is subscribed via `_mainVm.PropertyChanged += OnMainViewModelPropertyChanged`. This fires SYNCHRONOUSLY (WPF PropertyChanged is synchronous). So `LoadCurrentPageAsync()` is called synchronously within `SetCanvasSize`, but the async body is deferred.

Between `SetCanvasSize` completing and the first user click, the `_canvasWidthPx` / `_canvasHeightPx` are already updated. `EnsureCoordinateMapper` at click time uses them correctly.

REVISED ROOT CAUSE: The bug is in `WireViewModel`. When `DataContext` changes (PDF canvas view model arrives), `WireViewModel` calls `ReportCanvasSize()` using `ActualWidth` / `ActualHeight`. But `ActualWidth`/`ActualHeight` may be in LOGICAL pixels and `dpi.PixelsPerDip` may be 1.0 at that time (DPI hasn't been queried yet from the correct visual). `VisualTreeHelper.GetDpi(this)` returns 1.0 when the control is not yet in the visual tree. So `_canvasWidthPx` = logical width, and `_dpiScale` = 1.0. The mapper is created with `dpiScale=1.0` at this point.

Then `SizeChanged` fires → `ReportCanvasSize(e.NewSize)` → `VisualTreeHelper.GetDpi(this)` NOW returns the real DPI (e.g. 1.5) → `SetDpiScale(1.5)` then `SetCanvasSize(widthPx, heightPx)` with correct physical pixels.

At this point, `SetCanvasSize` calls `LoadCurrentPageAsync`. The bitmap loads async. `Paint()` fires. `EnsureCoordinateMapper()` calls `Update()` with `_dpiScale=1.5`, correct canvas size. Mapper is correct. User's FIRST click: `EnsureCoordinateMapper()` → `Update()` with `_dpiScale=1.5`. CORRECT.

I cannot reproduce the exact bug path from code reading alone. The definitive fix is to force `EnsureCoordinateMapper()` to be called (on the UI thread, synchronously) inside `SetCanvasSize` and `SetDpiScale`, AFTER updating the fields, so the mapper is always up-to-date before any click can fire. This is the defensive fix that eliminates all possible ordering races.

Purpose: Ensure `CoordinateMapper` is always synchronously current before the first user click at any zoom/DPI/canvas-size state.
Output: Modified `PdfCanvasViewModel.cs` with `EnsureCoordinateMapper()` called at the end of `SetCanvasSize` and `SetDpiScale`.
