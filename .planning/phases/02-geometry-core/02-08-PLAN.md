<objective>
Fix two hover-state visibility bugs in RightRail button styles.

GAP-8: The Delete button's danger-red background (#CC2020) is overridden by the RailButtonStyle ControlTemplate's IsMouseOver trigger, which sets BrushSurface2 (cream) directly on the Border named "Bd". Since TemplateBinding is used for the normal state, the inline Background="#CC2020" on the Button element sets the Button.Background dependency property — but the ControlTemplate trigger bypasses this by directly targeting the Border's Background, making white text invisible on cream.

Fix: Introduce DeleteButtonStyle (BasedOn=RailButtonStyle) with a custom ControlTemplate whose IsMouseOver trigger sets the Border Background to #991818 (dark red) instead of BrushSurface2.

GAP-9: StepButtonStyle has three triggers in this order:
  1. Tag="active" (sets cobalt)
  2. MultiTrigger IsMouseOver+Tag="active" (sets BrushAccentInk — darker cobalt on hover-while-active)
  3. MultiTrigger IsMouseOver (sets BrushSurface2 — generic hover)

WPF trigger precedence rule: within a ControlTemplate.Triggers collection, a later trigger has higher priority over the same property. Therefore trigger 3 (generic hover) overrides trigger 2 (hover-while-active), erasing the cobalt on hover.

Fix: Move the generic IsMouseOver MultiTrigger to position 2 and the IsMouseOver+Tag="active" MultiTrigger to position 3 (last = highest priority). The active+hover trigger must always appear AFTER the generic hover trigger in the collection.

Purpose: Readable hover states for all right rail buttons. Critical for gaze-driven use — button affordance must be visible at all times.
Output: Updated AppStyles.xaml with DeleteButtonStyle + corrected StepButtonStyle; updated RightRail.xaml Delete button using DeleteButtonStyle.
</objective>

**Change 2 — Add DeleteButtonStyle (GAP-8):**

After the StepButtonStyle closing tag, add a new style:

```xml
<!-- ── DeleteButton style: danger-red button with readable hover state ── -->
<!-- BasedOn RailButtonStyle but uses a custom ControlTemplate so IsMouseOver -->
<!-- sets a dark red (#991818) instead of cream — keeps white text readable. -->
<Style x:Key="DeleteButtonStyle" TargetType="Button" BasedOn="{StaticResource RailButtonStyle}">
    <Setter Property="Template">
        <Setter.Value>
            <ControlTemplate TargetType="Button">
                <Border x:Name="Bd"
                        Background="{TemplateBinding Background}"
                        BorderBrush="{TemplateBinding BorderBrush}"
                        BorderThickness="{TemplateBinding BorderThickness}"
                        CornerRadius="6">
                    <ContentPresenter HorizontalAlignment="Center" VerticalAlignment="Center"/>
                </Border>
                <ControlTemplate.Triggers>
                    <Trigger Property="IsMouseOver" Value="True">
                        <Setter TargetName="Bd" Property="Background" Value="#991818"/>
                    </Trigger>
                    <Trigger Property="IsPressed" Value="True">
                        <Setter TargetName="Bd" Property="Background" Value="#660000"/>
                        <Setter TargetName="Bd" Property="BorderBrush" Value="#CC2020"/>
                    </Trigger>
                    <Trigger Property="IsEnabled" Value="False">
                        <Setter Property="Opacity" Value="0.4"/>
                    </Trigger>
                </ControlTemplate.Triggers>
            </ControlTemplate>
        </Setter.Value>
    </Setter>
</Style>
```

The DeleteButtonStyle inherits Background, BorderBrush, Foreground, FontSize, FontFamily, Cursor, FocusVisualStyle from RailButtonStyle via BasedOn, but overrides the Template entirely so the hover trigger uses dark red (#991818) and the TemplateBinding Background is red (#CC2020) on normal state and pressed (#660000). No TargetName="Bd" setters exist in the base setters (they're in the Template) so this override is clean.
